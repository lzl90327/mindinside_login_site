# 《登录扩展能力口径冻结》

**文档版本**：v1.0（冻结草案供签认）  
**状态**：待产品、架构联合签认后生效  
**制定依据**：源系统 Web 管理端已实现行为与接口事实；本轮不引入与旧行为相冲突的默认变更。

---

## 1. 文档目的与适用范围

### 1.1 目的

本文件用于在**多系统统一登录页体系**（WMS、TMS、BMS、HR、CRM、AMS、统一业务平台）中，对以下**登录扩展能力**作出**书面冻结口径**：

- **记住账号密码**（与源系统「记住账号密码」文案及行为对齐）；
- **忘记密码 / 重置密码**（与源系统四阶段流程及接口契约对齐）。

本文件作为：**产品确认、架构确认、开发与测试验收、Figma/Cursor 排期的共同前置依据**。凡与本文件「冻结口径」冲突的实现或交互，须走**变更立项**与**书面修订本文件**，不得在未修订前提下以「更合理」为由默认替换。

### 1.2 适用范围

适用于上述七个系统在同一产品族内的 **Web 管理端登录及相关扩展流程**（与源系统分析范围一致）。  
**不包含**（除非另行立项并引用新附录）：小程序、员工端、货主端另一套 auth 前缀流程、设备指纹与设备审批等能力。

## 2. 能力范围

本冻结文档覆盖两类能力：

| 编号 | 能力名称 | 边界说明 |
|------|----------|----------|
| E-1 | **记住账号密码 / 记住密码** | 含勾选态、本地持久化字段与键名、失效策略、租户预填相关独立键、登录请求中的 `rememberMe` 传参行为；**不包含**后端在未闭环前对会话时长的隐含承诺（见 §4.6、§7）。 |
| E-2 | **忘记密码 / 重置密码** | 含四阶段用户路径、两接口调用次序与字段、前端各步完成条件、与后端一致的校验与频控语义（以接口与既有前端为准）。 |

## 3. 冻结原则

下列原则为本文件**最高优先级**，与任何「体验优化」「安全最佳实践」并列时，**以本节为准**（最佳实践仅可出现在 §7，且**不得反向修改** §4、§5 的冻结默认）。

1. **必须兼容旧行为**  
   对源系统已存在的「记住账号密码」「忘记密码/重置密码」相关行为，新体系在**当前阶段**须予以兼容，**不得自主更改**旧有定义。

2. **当前阶段不自主更改**  
   包括但不限于：不默认缩短流程、不默认合并步骤、不默认改变接口契约、不默认改变字段含义、不默认改变前端完成条件、不默认改变本地存储键名与内容结构（除非另行签认的「迁移专项」书面批准，且该专项不在本轮冻结范围内）。

3. **不默认优化、不默认升级**  
   「更合理」「更安全」「更短路径」等理由**不构成**本轮修改冻结口径的依据；若需变更，须**单独立项**并修订本文件版本。

4. **风险与演进单列**  
   凡安全、合规或产品层面的争议，均须在 **§7 风险与后续演进建议** 中记录；**§4、§5 冻结口径仍保持与旧行为一致**，不得以风险为由在本轮静默删改能力。

## 4. 记住账号密码能力冻结口径

### 4.1 当前旧行为定义（冻结为「应兼容目标态」）

1. **文案**  
   - 登录页展示文案为：**「记住账号密码」**（与源系统一致）。

2. **勾选后的本地持久化**  
   - 使用浏览器 `localStorage`。  
   - 存储键名：**`tms_saved_credentials`**。  
   - 存储内容（逻辑字段）：**`tenantId`、`username`、`password`**。  
   - `password` 经 **`obfuscate` 混淆**处理后再写入存储；**非加密**语义在实现上仍视为「可逆混淆」，但**本轮不改变**该实现策略的定义位置（仍属旧行为一部分）。

3. **时效**  
   - 超过 **30 天** 视为该条保存凭证**失效**（与旧逻辑一致）。

4. **未勾选行为**  
   - 未勾选「记住账号密码」时，**清除**上述保存项（与旧逻辑一致）。

5. **租户预填（独立来源）**  
   - 存在 **`last_tenant_id`**（或等价命名以代码事实为准），**即使未勾选**「记住账号密码」，仍可能作为**租户 ID 预填**来源；该行为与「保存凭证」策略**可并存**，**本轮不合并、不删减**该独立机制。

6. **登录请求**  
   - 前端在登录时向 **`/v1/auth/login`** 请求体携带 **`rememberMe`** 字段（与旧行为一致）。  
   - **已知事实**：后端主登录链**当前未消费** `rememberMe`；refresh token 默认时长等**不因**该字段在服务端形成闭环。**本轮冻结**：**前端传参行为保持不变**；服务端是否扩展消费**不属于**本轮「旧 Web 行为兼容」的必改项，但若未来变更须另文约定且不默认同步删减前端字段。

### 4.2 新系统当前阶段是否必须兼容

**是。** 新体系统一登录组件/页面在启用本能力时，**必须**兼容 §4.1 所述旧行为。

### 4.3 兼容程度（冻结）

- **UI 文案**：须为 **「记住账号密码」**（允许同义微调仅当产品书面批准且不与「仅记住账号」混淆；**默认不允许**微调）。  
- **存储键名与字段集合**：**`tms_saved_credentials`** 内 **`tenantId`、`username`、`password`（混淆后）`** 须可互操作或等价迁移（若多系统共存同一浏览器，以**不破坏旧键读写语义**为验收底线）。  
- **30 天失效、未勾选清除、`last_tenant_id` 独立预填**：**须一致**。  
- **`rememberMe` 传参**：**须继续传递**，语义与旧前端对齐（布尔或与旧请求体一致的具体类型以代码事实为准，**不得在本轮擅自省略该字段**）。

### 4.4 UI 文案表述（冻结）

- 主文案：**记住账号密码**  
- 不在本轮冻结中引入替代主文案（例如「记住账号」）作为默认替换。

### 4.5 前端行为定义（冻结）

1. 用户勾选并登录成功后：按 §4.1 写入 `localStorage`。  
2. 用户未勾选：清除 `tms_saved_credentials`（或旧逻辑等价清除范围）。  
3. 进入登录页时：按旧逻辑回填 `tenantId`、`username`、`password`（经反混淆可用性以旧行为为准）。  
4. `last_tenant_id`：**保留**既有独立预填行为，**不与**「是否勾选记住账号密码」强行绑定为互斥（与旧事实一致）。  
5. 登录请求：**携带 `rememberMe`**。

### 4.6 后端是否已有闭环（记录事实，不改变前端冻结）

- **事实记录**：主登录链**未消费** `rememberMe`；token / refresh **未**因该字段形成文档化闭环。  
- **冻结结论**：**不改变**前端 §4.5 行为；后端闭环若未来建设，须**单独版本**修订本文件 **§4.6 附录**，且**不得**在本轮以「无闭环」为由删除前端 `rememberMe` 或本地存储旧语义。

### 4.7 风险记录（不改变当前行为）

见 **§7.1**（本地混淆存密、`rememberMe` 与会话策略脱节等）。本节**仅引用**，**不构成**删减或替换 §4.1–4.5 的理由。

## 5. 忘记密码 / 重置密码能力冻结口径

### 5.1 当前旧流程步骤（冻结）

源系统 Web 管理端为 **4 个阶段**，阶段标识与顺序如下：

1. **`input`**：输入 `tenantId`；选择 `phone` 或 `email`；输入 `target`；调用 **`POST /v1/auth/forgot-password/send-code`**。  
2. **`verify`**：输入验证码；**前端仅校验验证码长度**；**不调用**后端校验接口作为该步完成条件（与旧行为一致）。  
3. **`reset`**：输入新密码；调用 **`POST /v1/auth/forgot-password/reset`**。  
4. **`done`**：成功页，返回登录。

### 5.2 新系统当前阶段是否必须完全兼容

**是。** 新体系须提供与上述 **四阶段、同顺序、同阶段内完成条件** 一致的用户路径；**不得**在本轮以「体验」为由默认合并、删减或重排阶段。

### 5.3 哪些步骤必须保持一致（冻结）

- 阶段数量：**4**（`input` → `verify` → `reset` → `done`）。  
- 各阶段内**用户输入项**及**触发接口的时机**与旧版一致：  
  - `input` 结束须调用 **send-code**；  
  - `verify` **不以**后端校验为通过条件（**仅长度**）；  
  - `reset` 须调用 **reset**；  
  - `done` 为成功收口并返回登录。

### 5.4 哪些接口契约必须保持一致（冻结）

| 接口 | 方法 | 冻结要点 |
|------|------|----------|
| 发送验证码 | `POST /v1/auth/forgot-password/send-code` | 字段：`type: 'phone' \| 'email'`、`target: string`、`tenantId?: number`；按 **`tenantId` + 手机/邮箱** 联合定位用户；验证码 **TTL 5 分钟**；频控：**同标识 60s**、**每日每号 10 次**、**每 IP 每小时 20 次**（与后端一致）。 |
| 重置密码 | `POST /v1/auth/forgot-password/reset` | 字段：`type`、`target`、`code`、`newPassword`、`tenantId?`；校验验证码后按租户查用户；密码策略 **8–32 位、须含字母与数字、拒绝弱密码**（与后端一致）。 |

**路径、字段名、枚举取值、可选 `tenantId` 语义**：均以旧系统契约为准，**本轮不修改**。

### 5.5 哪些交互条件必须保持一致（冻结）

- **`verify` 步骤**：通过条件为**前端对验证码长度的判断**；**不要求**在该步调用后端校验接口（与旧行为一致）。  
- **错误处理、loading、具体长度数值**：以实现侧**对齐旧前端事实代码**为验收依据；本文件不虚构数值，但**禁止**在本轮将「必须调后端校验」写为默认冻结（该属演进，见 §7）。

### 5.6 是否允许当前阶段改成更短流程

**不允许自主更改。**  
在未签认新版《登录扩展能力口径冻结》并完成变更立项前，**不得**将四阶段改为更少步骤或调换「先校验后重置」的完成条件定义。

### 5.7 后续若要优化，应以什么条件重新立项

须同时满足：

1. **书面修订**本冻结文档（版本号递增，变更说明含「旧行为 → 新行为」对照表）。  
2. **产品 + 架构**双签（必要时含安全/合规）。  
3. **兼容与迁移策略**单独立项（含：存量用户、书签/深链、自动化测试、回滚）。  

未满足以上条件，任何「缩短流程」「verify 增加后端校验」等均视为**范围外演进**，不得替代本轮默认。

## 7. 风险与后续演进建议

> **声明**：本节全部内容**不构成**对 §4、§5 的修改理由；**不授权**在未修订本文件前提下实施变更。

### 7.1 风险提示（记住账号密码）

- **本地混淆非加密**：`password` 以可逆混淆存于 `localStorage`，在终端被攻破或恶意扩展场景下存在泄露风险。  
- **`rememberMe` 与后端会话策略脱节**：用户可能误解勾选与「长期登录」关系；与 refresh **7d** 等事实并存时易产生**产品预期**问题。  
- **`last_tenant_id` 与凭证分离**：多租户场景下预填与凭证组合可能带来误操作风险（与旧行为一致，仅提示）。

### 7.2 风险提示（忘记密码）

- **`verify` 仅长度校验**：与后端最终校验存在**时间差**；恶意或错误输入在 UI 层可能较晚暴露（与旧行为一致，仅提示）。

### 7.3 后续演进建议（须单独立项，见 §5.7）

- 评估 **密钥保护**、**仅记住账号**、**服务端 rememberMe 闭环** 等方案。  
- 评估 **verify 步骤** 增加后端校验或统一错误模型的可行性。  
- 评估 **四阶段 UI** 在**新文件版本**下的合并与无障碍优化。
